# 이번 과제에서 이해하고 가야 할 내용

이번 B3-1 과제의 핵심은 단순히 Redis 명령어를 흉내 내는 것이 아니라, **Redis가 빠르게 동작하는 기반 자료구조를 직접 구현하고 연결하는 것**입니다.

## 1. 반드시 이해해야 할 세 가지 자료구조

### 해시맵: 데이터 저장소

키와 값을 저장하는 Mini Redis의 중심 자료구조입니다.

- `dict`를 사용하지 않고 직접 구현
- 직접 만든 해시 함수로 키를 버킷 인덱스로 변환
- 해시 충돌은 **체이닝**으로 해결
- 로드 팩터가 `0.75`를 넘으면 버킷 크기를 2배로 늘리고 재해싱
- 평균적으로 `put`, `get`, `remove`가 O(1)

핵심 질문은 다음과 같습니다.

> 서로 다른 키에서 같은 해시 인덱스가 나오면 어떻게 저장하고 찾는가?

### 이중 연결 리스트: LRU 순서 관리

최근에 사용된 키와 오래 사용되지 않은 키의 순서를 관리합니다.

- 리스트 앞쪽: 가장 최근에 사용된 키, MRU
- 리스트 뒤쪽: 가장 오래 사용되지 않은 키, LRU
- `SET` 또는 성공한 `GET`이 발생하면 해당 키를 맨 앞으로 이동
- 메모리가 초과되면 리스트 뒤의 키부터 제거
- 각 노드는 `prev`, `next`, `data`를 가짐

이중 연결 리스트만 사용하면 특정 키의 노드를 찾는 데 O(n)이 걸립니다. 따라서 다음처럼 결합해야 합니다.

```text
해시맵: key → LRU 노드
이중 연결 리스트: 실제 사용 순서 관리
```

이 조합 덕분에 특정 키의 노드를 O(1)에 찾고, O(1)에 맨 앞으로 옮길 수 있습니다.

### 최소 힙: TTL 만료 관리

TTL은 만료 시간이 가장 가까운 키를 빠르게 찾는 문제입니다.

힙에는 다음 형태의 데이터를 저장합니다.

```text
(expire_at, key)
```

최소 힙의 루트에는 항상 가장 빨리 만료될 항목이 위치합니다.

- `push`: O(log n)
- `pop`: O(log n)
- `peek`: O(1)
- `_heapify_up`, `_heapify_down` 직접 구현

TTL이 다시 설정되거나 키가 삭제되면 힙에 오래된 정보가 남을 수 있으므로, **lazy deletion 전략**도 이해해야 합니다. 힙에서 항목을 꺼낼 때 현재 TTL 정보와 일치하는지 확인하고, 오래된 기록이면 무시하는 방식입니다.

## 2. 전체 구조에서 관리할 상태

Mini Redis는 개념적으로 다음 상태를 함께 관리해야 합니다.

```text
데이터 해시맵       key → value
LRU 연결 리스트    최근 사용 순서
LRU 노드 해시맵    key → 연결 리스트 노드
TTL 해시맵          key → 현재 expire_at
TTL 최소 힙         (expire_at, key)
used_memory         현재 데이터 메모리 사용량
maxmemory           최대 메모리 제한
evicted_keys        LRU로 제거된 누적 키 개수
```

어떤 키를 삭제할 때는 한 자료구조에서만 지우면 안 됩니다. 데이터, LRU, TTL 상태가 서로 일관되도록 함께 정리해야 합니다.

## 3. 메모리와 LRU 처리

메모리 사용량은 실제 Python 객체 크기가 아니라 다음 공식만 사용합니다.

```text
used_memory =
Σ(UTF-8로 인코딩한 key 바이트 수 + value 바이트 수)
```

한글처럼 UTF-8에서 여러 바이트를 사용하는 문자열이 있으므로 `len(text)`가 아니라 다음 방식이 필요합니다.

```python
len(text.encode("utf-8"))
```

`SET` 처리의 중요한 흐름은 다음과 같습니다.

1. 새 키와 값의 크기를 계산
2. 기존 키라면 기존 크기를 빼서 예상 메모리를 계산
3. 단일 엔트리 자체가 `maxmemory`보다 크면 저장하지 않고 OOM 반환
4. 값을 저장하고 LRU 맨 앞으로 이동
5. 기존 키를 덮어썼다면 기존 TTL 제거
6. 전체 메모리가 제한을 넘으면 LRU 키부터 반복해서 제거
7. 제거할 때마다 `used_memory` 감소 및 `evicted_keys` 증가

특히 **덮어쓰기의 메모리 차이 계산**과 **UTF-8 바이트 계산**이 자주 실수하는 부분입니다.

## 4. TTL의 의미와 반환값

TTL은 단순히 만료 시각을 저장하는 데서 끝나지 않습니다. 키를 다루는 명령을 실행하기 전에 현재 시각과 비교해 만료 여부를 확인해야 합니다.

- `EXPIRE key seconds`
  - 키 없음: `(integer) 0`
  - 설정 성공: `(integer) 1`
  - `seconds <= 0`: 즉시 삭제 후 `(integer) 1`
- `TTL key`
  - 키 없음 또는 이미 만료: `(integer) -2`
  - 키는 있지만 TTL 없음: `(integer) -1`
  - TTL 있음: 남은 초 반환

만료된 키를 발견하면 다음 상태를 함께 정리해야 합니다.

```text
데이터 + LRU + TTL 정보
```

TTL로 삭제된 키는 LRU eviction이 아니므로 `evicted_keys`를 증가시키면 안 됩니다.

## 5. 명령어별 중요한 부수 효과

| 명령어 | 핵심 동작 | LRU 변경 | TTL 변경 |
|---|---|---:|---:|
| `SET` | 값 저장/덮어쓰기 | 맨 앞으로 이동 | 기존 TTL 삭제 |
| `GET` | 값 조회 | 성공했을 때만 맨 앞으로 이동 | 만료 확인 |
| `DEL` | 키 삭제 | 노드 삭제 | TTL 삭제 |
| `EXISTS` | 존재 확인 | 요구사항상 갱신 없음 | 만료 확인 |
| `DBSIZE` | 키 개수 | 없음 | 만료 키 처리 고려 |
| `KEYS` | 전체 키 출력 | 없음 | 만료 키 처리 고려 |
| `EXPIRE` | 만료 시간 설정 | 없음 | 힙과 TTL 정보 갱신 |
| `TTL` | 남은 시간 조회 | 없음 | 만료 확인 |

`GET`으로 만료된 키를 발견한 경우에는 먼저 삭제하고 `(nil)`을 반환하며, LRU를 갱신하지 않습니다.

## 6. CLI와 에러 처리

프로그램은 다음과 같은 REPL 형태여야 합니다.

```text
mini-redis> SET name "Alice"
OK
mini-redis> GET name
"Alice"
```

필요한 파싱 기능은 다음과 같습니다.

- 명령어는 대소문자를 구분하지 않도록 처리하는 것이 적절함
- 공백 없는 값 지원
- 큰따옴표로 감싼 공백 포함 값 지원
- `exit`, `quit`으로 종료
- 알 수 없는 명령, 인자 개수 오류, 정수 변환 오류 구분

Python에서는 따옴표 파싱을 위해 `shlex.split()`을 사용할 수 있습니다. 이는 Key-Value 저장소를 대신하는 컬렉션이 아니므로 과제의 `dict` 금지 취지와 충돌하지 않습니다.

## 7. 구현 시 가장 중요한 설계 원칙

각 자료구조는 독립된 모듈로 분리해야 합니다. 예를 들면 다음과 같습니다.

```text
doubly_linked_list.py
hash_map.py
min_heap.py
mini_redis.py
main.py
```

그리고 자료구조 자체와 Redis 정책을 구분하는 것이 좋습니다.

- 연결 리스트: 노드 삽입·삭제·이동만 담당
- 해시맵: 키 저장과 조회만 담당
- 최소 힙: 우선순위가 가장 작은 항목 관리
- Mini Redis: LRU, TTL, 메모리 정책을 조합
- CLI: 입력 파싱과 출력 담당

## 한 문장으로 정리

이 과제의 본질은 다음 구조를 직접 만드는 것입니다.

```text
해시맵으로 O(1) 데이터 접근
+ 이중 연결 리스트로 O(1) LRU 갱신
+ 최소 힙으로 가장 빠른 TTL 만료 탐색
+ 모든 구조의 상태를 일관성 있게 동기화
```

구현 난이도가 가장 높은 부분은 개별 자료구조보다는 **키가 덮어써지거나, 만료되거나, 삭제되거나, LRU로 제거될 때 모든 관련 구조와 `used_memory`를 정확히 함께 갱신하는 것**입니다.
